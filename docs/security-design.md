## 1. セキュリティ設計方針

### 1.1 基本原則
- **最小権限の原則**: 必要最低限のアクセス権限のみ付与
- **多層防御**: ネットワーク・OS・アプリの3層でセキュリティを担保
- **ゼロトラスト**: すべてのアクセスを検証

### 1.2 脅威モデル
| 脅威 | 対策 |
|------|------|
| 不正アクセス | セキュリティグループで送信元IP制限 |
| DBへの直接攻撃 | RDSをプライベートサブネットに配置 |
| 通信の盗聴 | HTTPS化（ACM + ALB） |
| 認証情報の漏洩 | 環境変数で管理、コードに直書き禁止 |
| DDoS攻撃 | ALB + AWS Shield Standard |

## 2. ネットワークセキュリティ

### 2.1 セキュリティグループ設計（最小権限）

#### SG-ALB
```
インバウンド:
  HTTP:80   from 0.0.0.0/0  （一般公開）
  HTTPS:443 from 0.0.0.0/0  （将来実装）
アウトバウンド:
  ALL from 0.0.0.0/0
```

#### SG-EC2
```
インバウンド:
  SSH:22    from マイIP のみ （管理者のみ）
  TCP:5000  from SG-ALB のみ （ALB経由のみ許可）
アウトバウンド:
  ALL （外部APIアクセスのため）
```

#### SG-RDS
```
インバウンド:
  MySQL:3306 from SG-EC2 のみ （EC2からのみ許可）
アウトバウンド:
  なし （DB→外部通信は不要）
```

### 2.2 NACL設計（将来実装）
- デフォルトNACLを使用（全許可）
- 本番環境ではカスタムNACLで追加制限

## 3. データセキュリティ

### 3.1 保存データの暗号化
| 対象 | 暗号化方式 | キー管理 |
|------|----------|---------|
| RDS | AES-256（AWS管理） | KMS |
| EBS | AES-256（AWS管理） | KMS |
| S3 | SSE-S3 | AWS管理 |

### 3.2 通信の暗号化（将来実装）
- ALB: ACMで SSL証明書発行
- HTTP → HTTPS リダイレクト設定
- EC2-RDS間: SSL接続（MySQL SSL）

## 4. アクセス管理

### 4.1 IAM設計方針
- ルートアカウントの日常利用禁止
- MFA（多要素認証）の有効化
- IAMロールで最小権限付与

### 4.2 EC2のIAMロール（将来実装）
```
ロール名: portfolio-ec2-role
ポリシー:
  - CloudWatchAgentServerPolicy（ログ送信用）
  - S3ReadOnlyAccess（静的ファイル読み込み用）
```
